# DIFY-style Workflow Engine — 合成设计方案 (build blueprint)

## Recommended Architecture
WINNER = Design 1 (Minimal MVP that ships fast) as the SCOPE/PHASING spine, with Design 2's NodeExecutor plugin-registry as the engine's internal dispatch contract grafted in. Rationale, grounded in source I read: every design's "agent node" reuses run_invocation() verbatim (confirmed app/services/invocation.py:27 `run_invocation(request: AgentRunRequest) -> AgentRunResult`; same call delegate.py:90-100 makes), so the differentiator is (a) which one correctly gates on the real bugs and (b) extensibility cost. Design 1 is the only one that elevates the tool-bridge bug to a hard P0 prerequisite, and Design 2 has the only true open/closed node contract (a line-for-line clone of app/tools/base.register_handler at base.py:52-64, registered via the import-side-effect block main.py:19-22). Adopt Design 1's deferral list (no loops/sub-workflows/code-sandbox/SSE in v1) but make node dispatch a tiny `register_node_executor/get_node_executor` registry from day one — it costs ~10 lines and lets deferred node types land later without touching the BFS traversal core.

Concrete shape: a WorkflowDefinition is a DAG of typed nodes + edges persisted as one JSON file via the existing registry pattern. A new async `run_workflow(wf, inputs)` in app/services/workflow_executor.py does Kahn ready-set traversal over a per-run variable store keyed by node_key, resolves {{node.field}} into each node's inputs before it runs, dispatches by node.type to a NodeExecutor, writes the node's output back, follows edges. Agent node = run_invocation() unchanged. Tool node = app.tools.base.get_handler(tool_code).handle(ToolContext(...)) (the exact path deep.py already uses). New versioned endpoints under /api/v1/agentapp/workflow/* are purely additive (main.py:46-50 gets one more include_router). Team management is fixed by promoting the already-smuggled fields (data/teams/comic-drama-team.json already carries leaderEmployeeKey + description via RegistryEntity extra:'allow', registry.py model line 16) into the schema and adding optional default_workflow_key; a Team AUGMENTS rather than replaces — its real pipeline becomes a seed WorkflowDefinition, delegation stays as the ad-hoc fallback.

TWO CONFIRMED PREREQUISITE BUGS (both block agent-nodes-with-tools, both verified): (1) runner.py:25 declares module-level tool_handler_registry, read at runner.py:52, but NOTHING ever writes it — all real handlers register into base._handlers (base.py:49). So in the live loop every tool resolves to the _stub at runner.py:55 ('尚未注册处理函数'). (2) The loop invokes `await handler.handler(fn_args)` where handler.handler is Callable[[str],Awaitable[str]] (loop.py:22, loop.py:165), but base.py handlers expose `.handle(ctx: ToolContext)` (base.py:42). The P0 fix bridges both: in _build_tool_handler, fall back to base.get_handler(rt.tool_code) and wrap `async def _adapt(args:str)->str: h=get_handler(rt.tool_code); return await h.handle(ToolContext(rt.tool_code,args,employee_key,extra_context))`. KNOWN v1 LIMITATION to surface, not silently inherit: loop.py:105 `client.chat.completions.create(...)` is SYNC/un-awaited, so asyncio.gather fan-out of agent/LLM nodes does NOT truly parallelize and blocks the event loop — wrap each agent node body in asyncio.to_thread, or document fan-out as effectively sequential for LLM nodes in v1.

## Execution Engine
New file app/services/workflow_executor.py — `async def run_workflow(wf:WorkflowDefinition, inputs:dict) -> WorkflowRun`. Node dispatch via a tiny registry app/services/workflow_nodes.py (register_node_executor/get_node_executor, mirroring base.py:52-64) so node types are pluggable without editing the traversal.

TRAVERSAL (Kahn ready-set, supports parallel fan-out + single-condition branch): build adjacency + indegree from edges; VALIDATE on entry (exactly one start, all edge endpoints exist, acyclic via Kahn — reject cycles -> status=failed with clear error; this validation also runs server-side on save). Seed variables={'start':inputs}. Loop: a node is runnable when all inbound edges whose source completed AND (for branch edges) whose source_handle matches the source's chosen handle are satisfied; run the whole ready batch via asyncio.gather. A condition node activates only ONE outgoing handle, so dead-branch nodes are never enqueued (surface as status=skipped). A join after a branch must still fire when one inbound edge was pruned — this diamond-graph skip-propagation is the easiest place to get the engine wrong; pin it with explicit unit tests. Guards: global asyncio.wait_for(run_timeout, default 600 — separate from agent.run_timeout since multi-node runs are longer) at the API layer, and a max_steps cap (default 50, WorkflowConfig) as cycle/runaway backstop.

VARIABLE STORE + {{node.field}}: helper resolve_template(text, variables) in app/runtime/template.py via re.sub(r'{{\s*([\w.]+)\s*}}', ...) splitting on first '.' into (node_key, field). agent/template/tool nodes publish {'output':<str>,'text':<same>}; tool nodes also publish parsed JSON keys when the result is a JSON object; start publishes raw input fields. Missing refs resolve to '' and are logged into the step (fail-soft, matches project fail-open default). NEVER use eval/exec — condition uses a SAFE comparator over the enumerated ops only (saved workflows are a stored-RCE vector otherwise).

PER NODE-TYPE:
- start: variables['start']=inputs.
- agent: render userInputTemplate -> user_input; req=AgentRunRequest(employee_key=cfg.employeeKey, user_input=rendered, workflow_key=wf.workflow_key  # FREE scope namespacing — AgentRunRequest.workflow_key (conversation.py:13) already feeds build_scopes (invocation.py:28); no design noticed this seam, structured_schema_json=cfg.structuredSchemaJson, extra_context=json.dumps({'__delegation_stack':[...], '__workflow_run_id':run_id, '__upstream':<resolved>})). `await run_invocation(req)` VERBATIM. Wrap in asyncio.to_thread IF true parallelism needed (because loop.py:105 OpenAI call is sync). Copy assistant_message->output, traces/token_usage->NodeStepResult. Threading __delegation_stack lets an agent node still delegate and keep respecting delegation_max_depth (delegate.py:36); DECIDE whether workflow hops count toward that depth (see open_decisions). NOTE: model_overrides on the request is a no-op today (invocation.py never reads it) — don't promise per-node model override in v1 unless invocation.py is extended.
- condition: evaluate cases in order via safe comparator; set activated source_handle to matching case.label or elseLabel.
- template: output = resolve_template(cfg.template).
- tool: resolve argsTemplate -> args_json; `h=get_handler(cfg.toolCode); output = await h.handle(ToolContext(cfg.toolCode, args_json, employee_key=''))` — exact path deep.py uses.
- knowledge: knowledge_retriever.search(employeeKey, query, topK) (dependencies.py:34); output=joined snippets.
- end: final_output = resolve_template(cfg.outputTemplate) or last node output.

ERROR POLICY: each node try/except; node.config.onError in {'stop'(default),'continue'}. 'stop' aborts run (status=failed, persist partial steps); 'continue' marks node failed and proceeds (fail-open option). LLM/delegation failures already return strings (run_invocation never raises for those), so they surface as node output. Persist WorkflowRun via WorkflowRunStore after each batch so the frontend can poll partial progress.

## MVP Node Set
- **start**: 入口节点：声明工作流输入字段(config.inputs)，把用户输入写入变量存储 variables['start']，作为 {{start.xxx}} 的来源
- **agent**: 核心多-subagent节点：包一个数字员工(employeeKey)，渲染 userInputTemplate 后原样调用现有 run_invocation()，并把运行中的 workflow_key 透传进 AgentRunRequest 复用 build_scopes 的作用域命名空间；输出 assistant_message 作为 {{node.output}} 向下游流转。多 subagent = 多个 agent 节点串/并联(这是把'LLM自主决定的delegate'换成'显式编排'的关键)
- **condition**: 分支节点：按 cases(var/op/value) 顺序用安全比较器(绝不用 eval)匹配，激活对应 source_handle 的出边实现 if/else；不匹配走 elseLabel；下游死分支标记 skipped，但分支汇合(join)节点在某条入边被剪枝时仍必须触发
- **template**: 纯文本拼装节点：用 {{node.field}} 把多个上游输出组合成新字符串，不调用 LLM、不耗 token(如把编剧+设计结果合并喂给生成节点)
- **tool**: 工具节点：复用 app.tools.base.get_handler(toolCode).handle(ToolContext(...))，即 deep.py 已在用的同款调用路径，argsTemplate 支持 {{}} 注入，零新工具代码即可暴露 builtin/strategy/knowledge 等现有工具
- **knowledge**: 知识检索节点：对指定员工知识库做 knowledge_retriever.search(employeeKey,query,topK)，把片段作为变量注入下游 agent 节点的提示词(把 RAG 步骤显式化)
- **end**: 出口节点：用 outputTemplate 汇总最终结果写入 WorkflowRun.final_output 返回给调用方

## Open Decisions
- **Canvas library: reactflow vs hand-rolled SVG**
  - options: ['Add reactflow ^11 (one new dep; industry standard; matches DIFY; pan/zoom/connect free)', "Hand-roll an SVG canvas (consistent with repo's 'custom-drawn components' ethos; no new dep; much more code)"]
  - recommendation: Add reactflow ^11. It is the only realistic way to ship a DIFY-grade canvas in P4's budget; keep node-config forms lib-agnostic so a future swap to hand-rolled SVG stays possible. One dep, not two — defer auto-layout (dagre/elkjs).
- **Does a Workflow REPLACE a Team, or AUGMENT it?**
  - options: ['Replace: Team becomes just a workflow (clean, but breaks delegate.py membership checks and existing flat-team behavior)', 'Augment: Team keeps member_employee_keys as source of truth + gains optional default_workflow_key; workflow is the explicit flow, delegation stays as ad-hoc fallback']
  - recommendation: Augment. member_employee_keys is load-bearing for delegate.py:66 and back-compat; teams without a workflow must keep working exactly as today. Workflow is the configured path; delegation is the LLM-decided path; they compose.
- **Run API: synchronous request/response vs SSE streaming**
  - options: ['Sync POST returning final WorkflowRun (+ optional poll of /runs for partial progress) — simplest, no new infra', "SSE 'watch it run live' node lighting — true DIFY UX but net-new StreamingResponse plumbing AND undermined by the sync OpenAI call at loop.py:105 freezing the loop mid-node, plus prod-proxy buffering risk"]
  - recommendation: Sync + poll in v1; defer SSE to a later phase. The sync OpenAI call would make 'live' node updates stutter exactly during LLM nodes, so SSE buys little until that call is moved to AsyncOpenAI/to_thread. Persisting WorkflowRun after each node already enables a polling progress view.
- **Do workflow agent-node hops count toward delegation_max_depth (default 3)?**
  - options: ['Count them (safe ceiling, but a long configured pipeline of >3 agent nodes that each also delegate could abort mid-run)', "Don't count workflow edges; only LLM-decided delegate_to_employee hops increment the stack (workflow depth is bounded separately by max_steps)"]
  - recommendation: Don't count workflow edges toward delegation_max_depth; bound workflow size with max_steps instead, and let __delegation_stack track only true delegate hops within each agent node. Otherwise a legitimate 4+ stage pipeline fails for the wrong reason. Make this explicit in code + spec.

## Biggest Risks
- P0 tool-bridge fix changes LIVE single-agent behavior: tools silently stubbed today (strategy create/rollback, shell_execute) will start really executing. VERIFIED: runner.py:25 dict never populated, runner.py:55 _stub. Mitigate by confirming the existing approval/shell_execute guards still gate, and smoke-testing one plain /agent/run before shipping anything else.
- False parallelism: loop.py:105 client.chat.completions.create is SYNC/un-awaited, so asyncio.gather over agent nodes serializes AND blocks the event loop. Designs 2/3 flagged this; Design 1 did not. Wrap agent-node bodies in asyncio.to_thread (or migrate to AsyncOpenAI), else 'parallel subagents' is non-functional and long runs freeze the server.
- Token/cost multiplies: each agent node is a full run_invocation (its own 5-12 iteration loop). A multi-node workflow is multiplicative vs a single agent. Need the separate longer-but-bounded run_timeout (600s) + max_steps cap + per-node token totals surfaced in WorkflowRun so users see cost.
- Stringly-typed variable contract: agent nodes emit free-form text, so {{node.field}} beyond .output/.text is unreliable unless the employee is told (via structured_schema_json) to emit JSON. v1 should document 'use condition/template on .output' and lean on existing structured-schema support for typed handoffs.
- Diamond-graph skip-propagation: a join node after an if/else must STILL fire when one inbound branch was pruned, or the run hangs waiting on a skipped source. This is the single easiest place to get the engine subtly wrong — pin with explicit unit tests on branch->two paths->join.
- Stored-RCE via condition/code: condition MUST use a safe enumerated-op comparator, never Python eval/exec, because workflows are persisted and re-run. (This is also why the code/jsonpath node is deferred out of v1.)
- Persistence races + bloat: FileJsonRegistry.save is plain write_text (registry.py:101, NOT atomic) — WorkflowRunStore must copy scrape_store._atomic_write; cap history (_MAX=50) and truncate node outputs ([:2000] precedent at loop.py:178) so base64/long-text outputs don't bloat run files. File-JSON has no locking — acceptable for single-user MVP, flag before multi-user.
- TeamDefinition schema change + delegation back-compat: old team JSON lacks new fields (handled by extra:'allow'+None) BUT member_employee_keys must stay backfilled or delegate.py:66 membership checks break. Startup migration must be backfill-only (mirror migrate_employees at main.py:73), never clobber edited data.
- Process risk: project follows OpenSpec discipline; this is a sizable capability and should land as openspec/changes/add-workflow-engine with spec deltas (P6), not a raw code drop, to stay compliant.

## Winning Design (Design 1 — MVP, full detail)
### architecture_summary
A Workflow is a DAG of typed nodes connected by edges, persisted as a JSON file like every other registry entity. A new WorkflowExecutor (app/services/workflow_executor.py) performs a topological/BFS traversal over the graph, maintaining a per-run variable store keyed by node_key. Each node has a node-type executor; the central one — the "agent" node — simply builds an AgentRunRequest and calls the EXISTING app.services.invocation.run_invocation() unchanged, capturing its assistant_message + traces + token_usage. Variable interpolation ({{node.field}} / {{start.field}}) resolves a node's inputs from upstream outputs before the node runs. Branching is done by a "condition" node that evaluates simple predicates and activates only the matching outgoing edge label. The whole run is wrapped in asyncio with a global timeout and a max-steps guard mirroring the existing run_timeout_seconds / max_iterations conventions. A WorkflowRun record (status, per-node steps, variable snapshots, tokens) is persisted via the same atomic-write file-store pattern as scrape_store.py. New endpoints live under /api/v1/agentapp/workflow (CRUD) and /api/v1/agentapp/workflow/{key}/run + /runs (history) — additive, breaking nothing. Frontend adds a /workflows route with a React Flow canvas, a node config Drawer reusing Ant Design 5 + theme.ts, and a run/debug panel showing per-node status + outputs + traces. Team management is fixed by (a) adding real fields to TeamDefinition (leader_employee_key, description, ordered roles) so the leader/stages stop being frontend tag-guesses, and (b) making a Team optionally own a default Workflow so "团队协作流转" is an explicit configured graph, not LLM-only delegation. NOTE: a real wiring gap was found — the main loop's tool handlers currently resolve to stubs (see reuse_existing); fixing it is a prerequisite for Agent nodes whose employees use tools.

### execution_engine
New app/services/workflow_executor.py — async def run_workflow(wf:WorkflowDefinition, inputs:dict)->WorkflowRun.

Traversal: build adjacency from edges; identify the single start node (type=='start'); maintain `variables: dict[node_key]->dict(fields)` seeded with variables['start']=inputs. Execute in a BFS/topological order over REACHABLE+ACTIVATED nodes: a node becomes runnable when all its inbound edges that are 'active' have completed. Because a condition node only activates ONE outgoing handle, downstream nodes on dead branches are never enqueued (status=skipped if surfaced). v1 assumes an acyclic graph; a visited-set + max_steps cap (default 50, configurable WorkflowConfig.max_steps) prevents runaway / accidental cycles — exceeding it -> status=failed with a clear error.

Variable store + {{node.field}} resolution: a helper resolve_template(text, variables) does regex re.sub(r'{{\s*([\w.]+)\s*}}', ...) splitting on first '.' into (node_key, field). Field access: an agent/template/tool node publishes {'output': <string>, 'text': <same>} (and tool nodes also publish parsed JSON keys when the result is a JSON object, so {{classify.label}} works). start publishes its raw input fields. Missing refs resolve to '' and are logged into the step (fail-soft) — matches project 'fail-open' default.

Per node-type execution:
- start: variables['start']=inputs; output = first input or json of inputs.
- agent: render userInputTemplate -> user_input; req=AgentRunRequest(employeeKey=config.employeeKey, userInput=rendered, modelOverrides=config.modelOverrides, extraContext=json of upstream summary). Call await run_invocation(req) VERBATIM. Store assistant_message as output; copy traces/token usage into NodeStepResult. This is where multi-subagent actually happens: each agent node is a distinct employee, run independently, outputs piped forward by edges.
- condition: evaluate cases in order against resolved var; set the activated source_handle to the matching case.label (or elseLabel); only edges with that source_handle are 'active'.
- template: output = resolve_template(config.template).
- tool: resolve argsTemplate -> args_json; reuse the EXISTING handler via app.tools.base.get_handler(toolCode).handle(ToolContext(tool_code, args_json, employee_key='')) — same call path deep.py already uses; output=tool string result.
- knowledge: call knowledge_retriever (app.dependencies.knowledge_retriever) for employeeKey+query; output=joined snippets.
- end: final_output = resolve_template(outputTemplate) or last node output.

Concurrency / parallel branches: when multiple nodes are simultaneously runnable (a fan-out from one node to several agents), gather them with asyncio.gather so independent subagents run in parallel; a join node simply waits because it isn't runnable until all active inbound edges finished. v1 keeps this implicit (no explicit merge-strategy config).

Error handling: each node wrapped in try/except; on failure set step.status=failed,error=str(exc). Node-level policy config.onError in {'stop'(default),'continue'}: 'stop' aborts the run (status=failed) and persists partial steps; 'continue' marks node failed and proceeds (fail-open option). LLM/delegation failures already return strings (not exceptions) from run_invocation, so they surface as node output, not crashes.

Timeouts / max-steps: whole run wrapped asyncio.wait_for(run_workflow(...), timeout=WorkflowConfig.run_timeout_seconds (default 600, separate from agent.run_timeout_seconds since multi-node runs are longer)); per-node agent calls already bounded by run_invocation's own behavior; max_steps guard as above. On wait_for TimeoutError -> WorkflowRun.status='timeout', persist partial. Everything async; the executor never blocks the event loop (pure awaits).

### Design 2 node-executor registry (grafted)
FILE app/runtime/workflow/engine.py — async, node-type-agnostic.

VARIABLE STORE (app/runtime/workflow/variables.py):
- class VariableStore: holds `_data: dict[str, dict]` keyed by node id; plus reserved keys "start" (trigger input) and "env" (static). Each node writes its whole structured output dict under its id: store.put(node_id, output_dict).
- get_path("n1.text") → store._data["n1"]["text"]; missing path → "" (fail-soft) with a logged warning recorded into NodeRun.
- Resolution contract `{{node.field}}`: render_template(s: str) via a single compiled regex r"\{\{\s*([\w.]+)\s*\}\}". Scalars become str(value); a whole-string single placeholder ("{{n1.items}}") preserves the native type (list/dict) so Iteration/Aggregator can consume arrays. Nested config dicts/lists resolved recursively (resolve_config). This is THE inter-node data contract — every executor receives already-resolved inputs; no executor ever parses {{}} itself.

CONTRACT (app/runtime/workflow/base.py):
@dataclass class NodeContext: node: WorkflowNode; resolved_config: dict; store: VariableStore; run: WorkflowRun; engine_opts: WorkflowEngineOptions
@dataclass class NodeResult: output: dict = {}; next_handle: Optional[str] = None; token_usage=None; traces=[]; control: str = "continue"   # control ∈ continue/stop/error
class NodeExecutor(ABC): node_type: str; async def execute(self, ctx: NodeContext) -> NodeResult: ...
Registry mirrors app/tools/base.py exactly: _executors dict, register_executor(inst), get_executor(type), get_all_executors().

TRAVERSAL ALGORITHM (run_workflow(wf, trigger_input, opts) -> WorkflowRun):
1. Validate graph: exactly one start, ≥1 end, all edge endpoints exist, DAG acyclic (Kahn's algorithm; reject cycles UNLESS inside an Iteration subgraph). Build adjacency + indegree maps.
2. Seed VariableStore: store.put("start", trigger_input).
3. READY-SET scheduling (not naive topo-list, to get parallelism): a node is runnable when all incoming edges' sources are done AND at least one incoming edge was "activated" (branch selection). Maintain `pending`, `done`, `activated_edges`.
   Loop while ready nodes exist:
     - Collect all currently-ready nodes → run them concurrently with asyncio.gather (FAN-OUT PARALLELISM: diverging branches / parallel agents execute together).
     - Per node: resolve_config(node.config) → NodeContext → asyncio.wait_for(executor.execute(ctx), timeout=node.config.timeout or opts.node_timeout_s). On success store.put(node.id, result.output); append NodeRun; save run (progress checkpoint).
     - BRANCHING: result.next_handle picks which out-edges activate. Condition/Iteration set next_handle ("true"/"false"/"case_k"); other nodes activate ALL out-edges (handle None). Edges whose source_handle != chosen handle mark their target's branch as not-activated; a node with zero activated incoming edges after all its sources finish is status="skipped" (DIFY-style branch pruning).
4. GLOBAL GUARDS: opts.max_steps (default 100) counts executed nodes → abort status="failed" error="max steps exceeded"; opts.global_timeout_s wraps the whole run in asyncio.wait_for at the API layer (reuse app/api/agent.py's asyncio.wait_for+504 pattern). 
5. ERROR HANDLING per node via config.on_error ∈ {raise(default)|continue|default_value}: raise → run fails fast, mark downstream skipped; continue → node failed but engine proceeds (downstream sees empty output, fail-soft per project CLAUDE.md "default to fail-open"); default_value → output = config.default_output.
6. End node(s): collect their resolved config into run.final_output. Aggregate token_usage by summing NodeRun.token_usage (reuse AgentTokenUsage).

ASYNC/CONCURRENCY: whole engine is async; LLM/Agent nodes await run_invocation (already async). The OpenAI client call inside loop.py is sync (client.chat.completions.create) — acceptable as-is for v1; note as a known limitation (parallel Agent nodes serialize on that sync call). ITERATION node runs its sub-graph (or single child node) per array element via asyncio.gather with a concurrency semaphore (config.max_concurrency, default 3) to bound fan-out.

DEPTH/REENTRANCY: Agent nodes that internally call delegate_to_employee still respect settings.agent.delegation_max_depth (unchanged). Workflow nesting (a Workflow node calling another workflow) is intentionally OUT of v1 scope to avoid over-engineering.
